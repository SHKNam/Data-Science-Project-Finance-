# Architecture Decision Records

## 철학
데이터 분석 품질 최우선. 재현 가능한 분석 파이프라인. 단순하고 명확한 코드.

---

### ADR-001: 데이터 수집 방식 — requests 직접 호출 (주) + OpenDartReader (보조)
**결정**: requests로 직접 DART API 호출, OpenDartReader는 빠른 프로토타이핑 및 결과 교차 검증용
**이유**: 직접 호출이 API 구조 이해에 유리하고, rate limiting/캐시를 직접 제어 가능
**트레이드오프**: 초기 코드 작성량이 많지만, 유연성과 학습 효과가 큼

### ADR-002: 딥러닝 프레임워크 — PyTorch
**결정**: PyTorch 사용 (TensorFlow 대신)
**이유**: 동적 그래프로 디버깅 용이, 연구 친화적, Apple Silicon MPS 백엔드 지원
**트레이드오프**: TensorFlow 대비 프로덕션 배포 도구(TFServing 등)는 부족하지만, 본 프로젝트는 분석 목적이므로 무관

### ADR-003: 모델 선정 방식 — 전수 비교 후 최적 선정
**결정**: 각 Phase별 후보 모델을 전부 학습하고 동일 평가 지표로 리더보드 비교 후 최적 모델 선정
**이유**: 데이터 특성에 따라 최적 모델이 달라지므로, 사전 가정 없이 공정한 비교 보장
**트레이드오프**: 학습 시간이 길어지지만, 최적 모델을 놓치지 않음
